Storage

Storage that proves itself.

File storage is part of the protocol, not a service bolted on. sum chain records file metadata on chain and continuously challenges the nodes holding the data to prove they still have it.

How it works

When a file is stored through sum chain, its metadata and Merkle root are recorded on chain while the encrypted content is held by archive nodes. Encryption keys are managed through an on-chain registry, with X25519 keys derived from the same Ed25519 identities that sign transactions.

Archive nodes stake Koppa to participate. The protocol assigns replication, and uploaders fund a per-file fee pool that pays the nodes holding the data. A new archive node can call storage_getFundedFiles to discover exactly which files the network will pay it to store.

Proof of Retrievability

Storage claims are cheap; proofs are not. Every block, the chain issues retrievability challenges against stored files. The assigned archive node must answer with a valid Merkle proof within 50 blocks.

Fail a challenge and 5% of the node's stake is slashed. Holding the data and serving it is profitable; pretending to hold it is expensive. No trusted third party audits storage, the consensus itself does.

File lifecycle

PendingMetadata registered on chain, replication being established
ActiveFully replicated, fee pool funded, PoR challenges running
AbandonedFee pool exhausted, storage obligation released

Query it

The storage layer is fully inspectable over JSON-RPC:

# Files the network currently pays to store
{"jsonrpc":"2.0","method":"storage_getFundedFiles","params":[],"id":1}

# Metadata and on-chain ACL for a file
{"jsonrpc":"2.0","method":"storage_getAccessList","params":["0xabc..."],"id":1}

# Open challenges assigned to an archive node
{"jsonrpc":"2.0","method":"storage_getActiveChallenges",
 "params":["8zZ1pfbpUcAmoByWKYgJgiFZWpmhWQKJ4"],"id":1}

Full method reference in the JSON-RPC API docs.
