# SRC-818: Course Offering Token

> **Status:** Phase 0 draft — design baseline, pre-legal-review. No wire types, no executor, no activation gate exist yet. This document specifies intended behavior for review only.

## Overview

SRC-818 defines a **Course Offering** — one live class instance of a course: a specific term and section, taught by specific instructor(s), with an enrolled roster, assignments/exams, submissions, and grades.

**Core product invariant — stated up front:**

> **The assignment submission itself happens in SNIP. SUM Chain records an auditable submission *receipt*, not the raw submission.**
>
> `SubmitAssignment` / `SubmitExam` record a submission receipt on chain: `offering_id`, `assessment_id`, `student_commitment`, `submission_commitment`, `snip_ref`, `submitted_at_height`/timestamp, `attempt`. The chain record does **not** expose the student's wallet address, name, ID, raw work, or grade.

**Two-layer submission model:**

1. **SNIP content submission.** The student uploads/submits work to SNIP and grants the instructor/grader access via SNIP private sharing / ACL. Student identity is used *privately inside SNIP*, never as public SUM Chain state.
2. **Chain submission receipt.** SUM Chain records the *fact* that a submission commitment exists for a `(student_commitment, offering, assessment, attempt)`. **The student does not need to be the SUM Chain transaction sender.** The tx `from` MAY be an institution, sponsor, relayer, LMS service account, or other authorized submitter (SRC-201-style sponsored submission). The public chain sender is **not** treated as the student identity. Student authorization is proven by the linked SRC-812 enrollment credential, the SNIP submission authorization, and/or a student-scoped signature committed inside the SNIP/private payload — not by `sender == student`.

The same principle governs grading: `GradeSubmission` is a chain transaction recording a **grade commitment** and an encrypted-feedback `SnipRef` — never the raw grade or feedback text.

Every SRC-818 offering **must** reference a non-Archived/Deprecated [SRC-817](SRC-817.md) catalog entry via `catalog_id` (one catalog → many offerings). Authorization is delegated: students/enrollment via **SRC-812**, instructors/TAs via **SRC-882**. The shared glossary, privacy/SNIP model, `ContentAccessPolicy` definition, transaction envelope, fee/nonce open question, and consolidated legal/product questions are in the companion document [SRC-81X-EDUCATION-SUITE.md](SRC-81X-EDUCATION-SUITE.md).

## Design Principles

Privacy non-negotiables (v1, locked baseline):

1. **No raw submissions on-chain** — only `content_ref` (SnipRef) + `content_commitment`.
2. **No raw grades on-chain** — only `grade_commitment` + encrypted `feedback_ref`. Authoritative transcripts/credentials are the job of SRC-810/SRC-811, **not** SRC-818.
3. **No answer keys on-chain** — only `answer_key_commitment`; plaintext released via SNIP post-close.
4. **No student PII on-chain** — no name, school/student ID, email, DOB, contact info, government ID, or stable cross-system identifier, in any field/index/event/commitment input. **Indirect/linkable data is also in scope** (small-cohort timestamps, unique sequences). See companion §3.4 (FERPA PII & linkability rule).
5. **`student_commitment`, not raw address**, is the default identifier — and it MUST be per-offering/per-context scoped, salted, non-reversible, and non-reusable across courses/institutions/systems. **No public raw student address index in v1.** Cross-offering analytics via access-controlled off-chain private indexer only. See companion §3.5.
6. **Mandatory `ContentAccessPolicy`** for every `SnipRef` in chain state. No dangling content refs.
7. **No unbounded `Vec` in primary records.** Content items, assessments, enrollment links, submissions, and grades are separate CF rows; the `Offering` primary record holds only counters/roots.
8. **SRC-818 is not the authoritative gradebook.** Raw grades stay in SNIP / institutional systems; SRC-818 holds only grade commitments + audit refs; role-based access + audit logs are required for grade detail (companion §3.4).
9. **Enrollment existence / submission timestamps** are minimized, non-directly-identifying commitments only; small-cohort re-identification cases fall back to a privacy-preserving workaround (companion §3.5).
10. **Submission is a chain *receipt*, not the work; the chain sender is not the student.** The actual submission happens in SNIP. The `SubmitAssignment`/`SubmitExam` tx `from` is an authorized submitter (institution / sponsor / relayer / LMS service account) — never the student wallet, never treated as student identity. Student authorization is proven by the SRC-812 enrollment credential + `student_commitment` + (where required) a `student_auth_commitment` proven inside the private payload. SRC-201-style sponsored submission.

> **Phase-1 gating:** legal/privacy review is conditional approval. The FERPA model in companion §3.4–§3.6 must be explicitly accepted (with a recorded sign-off artifact) or each risky construct must have a documented workaround before Phase 1. `AccessAudience::IndividualStudent` is **provisional** — see companion §3.2 / §6 Q9.

## Data Model

### `Offering` (primary record — counters/roots only)

Stored one-per `offering_id` in `edu_offerings`.

```text
Offering {
    offering_id:       [u8;32]   // see Commitment Generation
    catalog_id:        [u8;32]   // REQUIRED ref to SRC-817 (non-Archived/Deprecated)
    term:              String    // public coordinate (e.g. "2026FA")
    section:           String    // public coordinate (e.g. "A")
    // Academic calendar — public, non-PII; bounds the default student
    // submission window and the default content access window.
    instruction_start_at:          Timestamp
    instruction_end_at:            Timestamp
    final_grade_submission_deadline: Timestamp
    owner:             Address    // department/institution admin (offering controller)
    instructor_count:  u32
    instructor_root:   [u8;32]    // root over InstructorBinding child rows
    content_count:     u32
    content_root:      [u8;32]    // root over ContentItem child rows
    assessment_count:  u32
    assessment_root:   [u8;32]    // root over AssessmentRef child rows
    enrollment_count:  u32        // count only — never a roster
    enrollment_root:   [u8;32]    // root over EnrollmentLink child rows
    status:            OfferingStatus
    policy_id:         Option<[u8;32]>   // optional SRC-803 hook, off by default v1
    created_at_height: BlockHeight
    updated_at_height: BlockHeight
    nonce:             u64
}

OfferingStatus { Draft=0, Active=1, EnrollmentClosed=2, Completed=3, Archived=4, Suspended=5, Cancelled=6 }
CourseRole     { InstitutionAdmin, Instructor, TeachingAssistant, Grader, Student, Auditor }
AssessmentKind { Assignment=0, Exam=1, Quiz=2, Project=3 }
SubmissionStatus { Submitted=0, Graded=1, Resubmitted=2, Voided=3 }
GradeStatus      { Provisional=0, Finalized=1, Revised=2 }
AssessmentStatus { Draft=0, Open=1, Closed=2, Graded=3 }
ContentKind      { Syllabus, LectureMaterial, Reading, Resource, Other }
```

```text

SubmissionRecord {            // CF edu_submissions — this is a RECEIPT, not the work
                              // key offering_id || assessment_id || student_commitment || attempt
    submission_id:        [u8;32]
    student_commitment:   [u8;32]      // scoped pseudonym; NOT the student wallet
    attempt:              u16
    snip_ref:             SnipRef      // student-OWNED SNIP object (the actual work)
    submission_commitment:[u8;32]      // commitment to the submitted work (a.k.a.
                                       // content_commitment); the work stays in SNIP
    submitted_at:         Timestamp
    submitted_at_height:  BlockHeight
    late:                 bool
    enrollment_ref:       [u8;32]      // SRC-812 credential proving student authorization
    student_auth_commitment: Option<[u8;32]>  // commitment over a student-scoped
                                       // signature / SNIP submission authorization,
                                       // proven inside the private payload — NOT a
                                       // chain signature by the student wallet
    submitter:            Address      // chain tx `from`: institution / sponsor /
                                       // relayer / LMS service account / authorized
                                       // submitter. NEVER treated as student identity.
    access_policy:        ContentAccessPolicy   // student + instructor + assigned graders
    status:               SubmissionStatus
}

GradeRecord {                 // CF edu_grades,
                              // key offering_id || assessment_id || student_commitment
    grade_id:          [u8;32]
    submission_id:     [u8;32]
    grade_commitment:  [u8;32]         // NEVER the raw grade
    feedback_ref:      Option<SnipRef>
    feedback_access:   Option<ContentAccessPolicy>   // student + instructor/grader only
    grader:            Address
    grader_role:       CourseRole
    graded_at_height:  BlockHeight
    status:            GradeStatus
    finalized:         bool
}
```

`SnipRef`, `ContentAccessPolicy`, `AccessAudience` canonical definitions: companion doc.

## `ContentAccessPolicy` application matrix

Every SnipRef above pairs with a policy. Default policies:

| Object | `AccessAudience` | opens_at | closes_at | grace_until | revoke_on_course_archive |
|---|---|---|---|---|---|
| Syllabus / lecture materials | `EnrolledStudents` | publish | none | n/a | true |
| Assignment instructions | `EnrolledStudents` | publish | none | n/a | true |
| Exam content | `EnrolledStudents` | exam open | exam close | optional | true |
| Answer key | `StaffOnly` → (post-close) `EnrolledStudents` | (key set) / (assessment close) | none | n/a | true |
| Submission | `IndividualStudent(student_commitment)` + instructor + assigned graders | submit | none | n/a | per retention policy |
| Feedback / grade detail | `IndividualStudent(student_commitment)` + instructor/grader | grade publish | none | n/a | per retention policy |

> **`IndividualStudent` rows are provisional.** If legal cannot confirm a per-student commitment in on-chain policy is FERPA-safe, the Submission/Feedback rows above are served by **SNIP-only ACL targeting** and the on-chain `ContentAccessPolicy` for those objects stays **audience-class-only** (`StaffOnly` + the student via SNIP ACL). Hard Phase-1 blocker — companion §3.2 / §6 Q9.

**Default academic access window.** Unless an object's policy overrides it, course-content access runs from the offering's `instruction_start_at` through `final_grade_submission_deadline`. Extensions/accommodations may extend access **per student or per assessment**. Archive/revoke behavior follows school policy (`revoke_on_course_archive`). This model is intended to be compatible with Canvas/Moodle-style LMS access behavior (content visible during the term, access tail through grade finalization, then archived/revoked per institution policy). The canonical default-window guidance is in [SRC-81X-EDUCATION-SUITE.md](SRC-81X-EDUCATION-SUITE.md) §3.2.

The chain stores the policy/schedule + commitments + refs. **SNIP enforces** actual private object access (encrypted ACL / key bundles deliver decryption to exactly the named audience within the time window). The chain never holds decryption keys.

## SNIP object-ownership model

- The **instructor controls publication and access policy** for course content (not "owns" it).
- **SNIP object ownership varies by object type:** syllabus / lecture materials / assignment instructions / exam content / answer keys → owned by **institution or instructor**; feedback / grade detail → owned by **instructor/grader**.
- **Submissions: the student owns/controls the submitted SNIP object.** `SubmitAssignment`/`SubmitExam` *grants* scoped access to the instructor and assigned graders per the course `ContentAccessPolicy`; it does **not** transfer ownership.

## Operations

Delivered as `EducationTxData { standard: CourseOffering, operation, data, recipient }`. Codes stable/append-only.

| Code | Operation | Actor | Key preconditions | Effect |
|---|---|---|---|---|
| 0 | `CreateOffering` | InstitutionAdmin | `sender==owner`; `catalog_id` resolves to Active/Deprecated SRC-817 entry | New `Offering` (Draft) |
| 1 | `UpdateOffering` | Admin / Instructor | status ∈ {Draft, Active} | Metadata edits |
| 2 | `PublishContent` | Instructor | SRC-882 teaching role; status ∈ {Draft, Active} | Add/update `ContentItem` + access policy |
| 3 | `AddAssessment` | Instructor | SRC-882 teaching role | Add `AssessmentRef` + access policies |
| 4 | `UpdateAssessment` | Instructor | assessment status ∈ {Draft, Open} | Adjust window/weight/policy |
| 5 | `OpenEnrollment` | InstitutionAdmin | status ∈ {Draft, Active} | Accept enrollment links |
| 6 | `CloseEnrollment` | InstitutionAdmin | status == Active | → `EnrollmentClosed` |
| 7 | `LinkEnrollment` | Student (self) / Admin | SRC-812 credential valid + subject-bound to sender | Bind `enrollment_ref`; store `student_commitment` |
| 8 | **`SubmitAssignment`** | **Authorized submitter** (institution / sponsor / relayer / LMS service account; **not necessarily the student wallet**) | see pipeline below | New `SubmissionRecord` (receipt) |
| 9 | **`SubmitExam`** | **Authorized submitter** (as above) | pipeline below; `kind==Exam`, within exam window | New `SubmissionRecord` (receipt) |
| 10 | `GradeSubmission` | Instructor/TA/Grader | SRC-882 grading role; submission exists | New `GradeRecord` (Provisional) |
| 11 | `FinalizeGrade` | Instructor | grade status == Provisional | `Provisional → Finalized` |
| 12 | `FinalizeCourse` | InstitutionAdmin | all grades Finalized | → `Completed` |
| 13 | `ArchiveOffering` | InstitutionAdmin | status == Completed | → `Archived`; trigger `revoke_on_course_archive` policies |
| 14 | `SuspendOffering` / `CancelOffering` | InstitutionAdmin | suspend reversible; cancel terminal | → `Suspended` / `Cancelled` |

### `SubmitAssignment` / `SubmitExam` validation pipeline (normative)

Ordered; fail-stop. **Fee/nonce on a failed step is an open chain-consistency question — companion doc §Fee/Nonce — and is NOT decided here.**

1. Education activation gate open, else reject.
2. Offering exists; `status ∈ {Active, EnrollmentClosed}`; rejected in `Draft`, `Suspended`, `Cancelled`, `Archived`.
   - `EnrollmentClosed` blocks only **new enrollment links**. Instructor/staff grading and coursework administration remain allowed regardless.
   - A **student submission** while the offering is `EnrollmentClosed` is permitted **only if an explicit per-assessment or per-student policy allows it** — extension, late window, incomplete, or a school-approved accommodation. Absent such a policy, `EnrollmentClosed` offerings reject new student submissions.
3. Assessment exists under offering; `kind` matches the operation; `status == Open`. Submission must fall within the **student submission window**:
   - within the assessment window `opens_at ≤ now ≤ due_at`; or
   - within grace `due_at < now ≤ grace_until` → set `late = true` (accept-and-flag; penalty computed off-chain); or
   - within an **explicit extension/accommodation window** granted for this student/assessment.
   - The default student submission window is governed by the assessment `opens_at`/`due_at`/`grace_until`, bounded by the offering's academic calendar (`instruction_start_at` … `final_grade_submission_deadline`). Extensions/accommodations may widen it per student or per assessment.
4. **The chain tx `sender` is the authorized submitter, NOT the student.** The sender is recorded as `submitter` and is never interpreted as the student identity. There is no `sender == student` check.
5. **Student authorization is proven without the student being the sender**, via: (a) the referenced `student_commitment` matching an existing `EnrollmentLink` on this offering; (b) a valid SRC-812 credential (`enrollment_ref`) — exists, non-revoked (SRC-805), in validity window, bound to that `student_commitment`, scoped to this offering/term; and (c) where required by policy, a `student_auth_commitment` over a student-scoped signature / SNIP submission authorization proven inside the private payload. None of these expose the student wallet on chain.
6. Attempt count for `(offering, assessment, student_commitment)` `< max_attempts`.
7. Transaction `data` carries only `snip_ref` + `submission_commitment` (+ `student_commitment`, `enrollment_ref`, optional `student_auth_commitment`); validator rejects raw/oversize payloads (tight max on `data`). No raw work, no student wallet, no PII.
8. Write the `SubmissionRecord` **receipt**, update child-row indexes, increment counters. Fee/nonce per the open chain-consistency decision (the *sponsor/submitter* pays any fee, not the student).

Distinct failure classes (gate-closed / offering-not-active / window-closed / not-enrolled / invalid-or-revoked-enrollment-credential / attempts-exhausted / raw-payload-rejected) so clients can react precisely. There is no "wrong-sender" class — the sender is an authorized submitter, not the student.

## Lifecycle

```
Draft ─Open─► Active ─Close─► EnrollmentClosed ─Finalize─► Completed ─Archive─► Archived
  │            │  ▲                  │                                   (read-only,
  │            │  └──reopen(admin)───┘                                    terminal)
  │            ▼
  └─► Cancelled (terminal)     Suspended ──resume──► (prior state)
```

| From | To | Operation | Notes |
|---|---|---|---|
| Draft | Active | `OpenEnrollment` | Enrollment accepting |
| Active | EnrollmentClosed | `CloseEnrollment` | Grading/admin still allowed; **student submissions only via explicit extension/accommodation policy** |
| EnrollmentClosed | Active | `OpenEnrollment` | Reopen pre-Completion |
| EnrollmentClosed | Completed | `FinalizeCourse` | Requires all grades Finalized |
| Completed | Archived | `ArchiveOffering` | Triggers `revoke_on_course_archive` |
| {Active, EnrollmentClosed} | Suspended | `SuspendOffering` | Blocks mutating ops; resumes to prior state |
| {Draft, Active} | Cancelled | `CancelOffering` | Terminal; records retained for audit |

Grading and coursework administration occur in `Active` and `EnrollmentClosed`. Student submissions occur freely in `Active`; under `EnrollmentClosed` they are accepted **only when an explicit extension/accommodation policy permits** (see the submit pipeline). `Archived`/`Cancelled` are terminal/read-only.

## Commitment Generation

```text
offering_id        = BLAKE3(b"SRC818-OFFERING:v1:" || catalog_id || term || section || creator || nonce_le)
student_commitment = BLAKE3(b"SRC818-STUDENT:v1:"  || subject || offering_id || salt)
content_commitment = BLAKE3(b"SRC818-SUBMISSION:v1:" || offering_id || assessment_id || student_commitment || attempt_le || work_hash || salt)
grade_commitment   = BLAKE3(b"SRC818-GRADE:v1:"    || offering_id || assessment_id || student_commitment || grade_value || salt)
spec_commitment    = BLAKE3(b"SRC818-SPEC:v1:"     || assessment_id || due_le || max_attempts_le || weight_le || salt)
```

Because `submitted_at_height`, `due_at`, and attempt counts are on-chain, **lateness and attempt-limit compliance are publicly verifiable without exposing the work**.

## Authorization

| Operation | Chain sender (`from`) | Student-identity check? | Source of authority |
|---|---|---|---|
| Create/Update/Open-Close-Enroll/Finalize/Archive/Suspend/Cancel | offering owner (`sender==owner`) | n/a | SRC-818 offering owner |
| PublishContent / Add-UpdateAssessment | instructor | n/a | SRC-882 (teaching role) |
| LinkEnrollment | student (self) **or** admin/relayer on behalf | no `sender==student` requirement | SRC-812 credential bound to `student_commitment` |
| SubmitAssignment / SubmitExam | **authorized submitter** (institution / sponsor / relayer / LMS service account) | **NO — sender is never treated as the student** | SRC-812 enrollment credential + `student_commitment` + (policy) `student_auth_commitment` proven in the private payload |
| GradeSubmission | instructor / TA / grader | n/a | SRC-882 (grading role) |
| FinalizeGrade | instructor | n/a | SRC-882 |

The public chain sender is **never** treated as the student identity. For submissions the sender is a sponsor/relayer/service account; student authorization is proven via credential + commitment + private-payload authorization (SRC-201-style sponsored submission). Soulbound by default; controller hand-off (if ever) is a future admin op, never a market transfer.

## Security Considerations

1. **Submission spam / DoS.** Mitigated by attempt limits + enrollment binding; residual risk depends on the open fee/nonce decision. If zero-fee-on-failure is chosen, mempool admission (Phase 3) must pre-check gate/offering-active/enrollment/window/attempts.
2. **Enrollment-binding integrity.** Submission requires a valid, non-revoked, subject-bound SRC-812 credential scoped to the offering.
3. **Late/attempt provability.** On-chain heights + counts make compliance publicly verifiable without exposing work.
4. **SNIP enforcement boundary.** Chain stores policy/commitments/refs only; confidentiality depends on SNIP encrypting to the policy-named audience. The chain cannot leak content it never holds.
5. **Append-only discipline.** No CF/enum reorder; additive evolution only.

## Example Usage (illustrative, design-level)

```text
1. CreateOffering { catalog_id: <SRC-817 CS101>, term:"2026FA", section:"A" } → Draft
2. PublishContent { ContentItem(Syllabus, SnipRef, policy=EnrolledStudents) }
3. AddAssessment { kind=Assignment, instructions_ref, opens_at, due_at, max_attempts:2 }
4. OpenEnrollment ; LinkEnrollment { enrollment_ref: <SRC-812 cred> } (student or relayer on behalf)
5a. (SNIP) Student uploads work to SNIP, grants instructor/grader ACL — student
    identity used privately in SNIP only.
5b. (Chain) Authorized submitter (institution/sponsor/relayer/LMS svc acct) sends
    SubmitAssignment { student_commitment, submission_commitment, snip_ref,
    enrollment_ref, student_auth_commitment? } → SubmissionRecord *receipt*
    (no student wallet, no raw work on chain; submitter pays any fee)
6. Instructor GradeSubmission { grade_commitment, feedback_ref (encrypted) } → Provisional
7. FinalizeGrade → FinalizeCourse → ArchiveOffering (revoke_on_course_archive fires)
```
