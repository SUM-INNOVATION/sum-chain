# SRC-817: Course Catalog Token

> **Status:** Phase 0 draft — design baseline, pre-legal-review. No wire types, no executor, no activation gate exist yet. This document specifies intended behavior for review only.

## Overview

SRC-817 defines the **reusable academic course identity** on SUM Chain: the abstract, institution-scoped definition of a course (e.g. "CS101 — Introduction to Computer Science, 3 credits, prerequisites X/Y") that many concrete class instances instantiate.

SRC-817 is one half of the education/LMS suite. The other half, [SRC-818](SRC-818.md), defines a **Course Offering** — one live class instance (a specific term/section, with an instructor, roster, assignments, submissions, and grades). Every SRC-818 offering **must** reference a non-Archived/Deprecated SRC-817 catalog entry via `catalog_id`. One catalog entry → many offerings.

The shared glossary, privacy/SNIP model, `ContentAccessPolicy` definition, transaction-envelope definition, and consolidated legal/product questions live in the companion document [SRC-81X-EDUCATION-SUITE.md](SRC-81X-EDUCATION-SUITE.md). This document is catalog-specific and references the companion for anything shared.

## Design Principles

1. **No PII on-chain.** Catalog coordinates (`department`, `course_code`, `term` semantics) are public-by-nature administrative identifiers, not personal data. Anything potentially sensitive or proprietary (course title, credit weighting for some institutions) is expressible as a BLAKE3 commitment.
2. **The chain stores verifiability, not content.** Descriptions, syllabi, learning outcomes, assessment policy, and accreditation documents live in SNIP; the chain holds only commitments + `SnipRef`s.
3. **Mandatory access policy.** Every `SnipRef` recorded in SRC-817 chain state MUST carry an associated `ContentAccessPolicy` (defined in the companion doc). No dangling content reference with unclear access rules is permitted.
4. **Append-only.** SRC-817 is delivered through the unified `TxPayload::Education` variant (see companion doc). No existing wire layout is altered.
5. **Versioned, not mutated.** Catalog entries evolve by supersession (new version, linked), not destructive edits to historical records.

## Data Model — `CatalogEntry`

Primary record. Stored one-per `catalog_id` in the `edu_catalog_entries` column family. **Holds only scalar fields, commitments, refs, and counters/roots — no unbounded `Vec` of child data** (see *Bounded-collection rule* below).

```text
CatalogEntry {
    catalog_id:              [u8;32]   // see Commitment Generation
    institution_id:          [u8;32]   // commitment to institution identity
    department:              String    // public administrative coordinate
    course_code:             String    // public administrative coordinate (e.g. "CS101")
    course_title:            Option<String>      // plaintext title (default)
    title_commitment:        Option<[u8;32]>     // for proprietary/confidential programs
    course_level:            CourseLevel
    credit_hours:            Option<u16>         // plaintext (default, verifiable downstream)
    credit_commitment:       Option<[u8;32]>     // for institutions treating credit as sensitive
    description_ref:         Option<SnipRefWithPolicy>
    learning_outcomes_ref:   Option<SnipRefWithPolicy>
    default_syllabus_ref:    Option<SnipRefWithPolicy>
    default_assessment_policy_ref: Option<SnipRefWithPolicy>
    prerequisites_root:      Option<[u8;32]>     // root/commitment over the prerequisite set
    prerequisites_count:     u32
    accreditation_root:      Option<[u8;32]>     // root/commitment over accreditation refs
    accreditation_count:     u32
    status:                  CatalogStatus
    version:                 u32
    supersedes:              Option<[u8;32]>     // prior catalog_id
    superseded_by:           Option<[u8;32]>     // next catalog_id
    owner:                   Address             // institution/registrar authority
    created_at_height:       BlockHeight
    updated_at_height:       BlockHeight
    nonce:                   u64
}

CourseLevel  { Undergraduate, Graduate, Doctoral, Professional, Continuing, Other }
CatalogStatus { Draft = 0, Active = 1, Deprecated = 2, Archived = 3 }
```

**Title / credit disclosure.** Exactly one of `course_title` / `title_commitment` is set; exactly one of `credit_hours` / `credit_commitment` is set. Default is plaintext (`course_title`, `credit_hours`) because downstream consumers (offerings, transcript standards) benefit from verifiable values. Commitment variants exist for confidential executive/proprietary programs.

**Bounded-collection rule (locked baseline).** `prerequisites` and `accreditation_refs` are **not** stored as inline `Vec`s on the primary record. They are child rows in dedicated CFs (`edu_catalog_prerequisites`, `edu_catalog_accreditation`), and the primary record holds only a count + a root/commitment. This prevents unbounded primary-record growth, consistent with the suite-wide rule.

### `SnipRefWithPolicy` (mandatory pairing)

Every off-chain reference in SRC-817 is the pair *(SnipRef, ContentAccessPolicy)* — never a bare `SnipRef`. The canonical definitions of `SnipRef`, `ContentAccessPolicy`, and `AccessAudience` are in the companion doc; SRC-817 only specifies its **default policies**:

| Catalog ref | Default `AccessAudience` | opens_at | closes_at | revoke_on_archive |
|---|---|---|---|---|
| `description_ref` | `Public` | on publish | none | false |
| `learning_outcomes_ref` | `Public` | on publish | none | false |
| `default_syllabus_ref` | `Public` | on publish | none | false |
| `default_assessment_policy_ref` | `StaffOnly` | on publish | none | false |
| `accreditation` (child rows) | `StaffOnly` | on publish | none | false |

Institutions may override defaults at publish time within policy.

## Operations

Delivered as `EducationTxData { standard: CourseCatalog, operation, data, recipient }` (envelope defined in companion doc). Operation codes are stable and append-only.

| Code | Operation | Actor | Preconditions | Effect |
|---|---|---|---|---|
| 0 | `CreateCatalogEntry` | Institution/registrar | `sender == owner`; unique `(institution_id, department, course_code, version)` | New entry, status `Draft` |
| 1 | `UpdateCatalogEntry` | Institution/registrar | `sender == owner`; status ∈ {Draft, Active} | Metadata edits (no version change) |
| 2 | `PublishCatalogContent` | Institution/registrar | `sender == owner`; status ∈ {Draft, Active} | Set/replace `*_ref` SnipRefs + their `ContentAccessPolicy` |
| 3 | `DeprecateCatalogEntry` | Institution/registrar | `sender == owner`; status == Active | `Active → Deprecated` |
| 4 | `SupersedeCatalogEntry` | Institution/registrar | `sender == owner`; new entry created | Set `supersedes`/`superseded_by` link; new version `Active` |
| 5 | `ArchiveCatalogEntry` | Institution/registrar | `sender == owner`; status ∈ {Deprecated, Active, Draft} | Terminal `Archived`, read-only |

Fee/nonce semantics for failed validation are an **open chain-consistency question** — see companion doc §Fee/Nonce. Not decided here.

## Lifecycle

```
Draft ──Update/Publish──┐
  │                     │
  │  (Activate via      ▼
  │   first publish) Active ──Deprecate──► Deprecated ──Archive──► Archived
  │                     │                     │                    (read-only,
  └────────Archive──────┴─────Archive─────────┘                     terminal)

Supersede: Active(vN) ──► creates Active(vN+1), links supersedes/superseded_by
```

Transition rules:

| From | To | Operation | Notes |
|---|---|---|---|
| Draft | Active | `PublishCatalogContent` / `UpdateCatalogEntry` | Entry becomes bindable by offerings |
| Active | Deprecated | `DeprecateCatalogEntry` | **Existing offerings remain valid; no NEW offerings may bind it** |
| Active / Draft / Deprecated | Archived | `ArchiveCatalogEntry` | Terminal, read-only |
| Active(vN) | Active(vN+1) | `SupersedeCatalogEntry` | New version; old version typically Deprecated |

**Deprecated semantics:** a deprecated catalog entry is still resolvable. SRC-818 `CreateOffering` against a Deprecated or Archived `catalog_id` is rejected; offerings already bound before deprecation continue to function.

## Commitment Generation

House-style domain-separated BLAKE3 (matches SRC-84X/88X conventions):

```text
catalog_id        = BLAKE3(b"SRC817-CATALOG:v1:" || institution_id || department || course_code || version_le || nonce_le)
title_commitment  = BLAKE3(b"SRC817-TITLE:v1:"   || institution_id || title_bytes || salt)
credit_commitment = BLAKE3(b"SRC817-CREDIT:v1:"  || institution_id || credit_le  || salt)
prerequisites_root  = merkle/BLAKE3 over the ordered set of prerequisite catalog_ids
accreditation_root  = merkle/BLAKE3 over the ordered set of accreditation SnipRef content_roots
```

Prerequisite-cycle prevention is an **off-chain validation responsibility** — the chain stores references only.

## Transaction Format

Unified suite envelope (canonical definition in companion doc):

```text
EducationTxData {
    standard:  EducationStandard::CourseCatalog,
    operation: u16,        // SRC-817 operation code
    data:      Vec<u8>,    // bincode of the operation struct
    recipient: Address,    // unused for catalog ops (set to owner); reserved
}
```

`Education` is `TxPayload` variant 22 / `TxType = 22` (append-only; see companion doc §Compatibility). No catalog operation mints a transferable token — catalog entries are non-transferable institutional records.

## Storage Schema

| Column family | Key | Value | Notes |
|---|---|---|---|
| `edu_catalog_entries` | `catalog_id` | `CatalogEntry` | Primary record |
| `edu_catalog_prerequisites` | `catalog_id \|\| prereq_catalog_id` | `()` | Child rows (bounded-collection rule) |
| `edu_catalog_accreditation` | `catalog_id \|\| idx` | accreditation `SnipRefWithPolicy` | Child rows |
| `edu_catalog_by_institution` | `institution_id \|\| catalog_id` | `()` | Index |
| `edu_catalog_by_code` | `BLAKE3(department\|\|course_code) \|\| catalog_id` | `()` | Dedupe / lookup |
| `edu_catalog_by_status` | `status_byte \|\| catalog_id` | `()` | Operational index |

No SRC-817 column family contains student data of any kind. Final CF naming/registration is **not** performed in Phase 0.

Audit log (commitments only, no PII):

```text
CatalogPublished  { catalog_id, ref_kinds_changed, height }

```

## Authorization

- **Catalog authority = `owner`** (institution/registrar address). All mutating operations require `sender == owner`.
- Optional SRC-803 policy hook (`policy_id`) for institutions wanting policy-driven trust — **off by default in v1**.
- Institution identity representation (opaque `institution_id` commitment vs. promotion to a first-class SRC-802 `Issuer` with stake/reputation/slashing) is an **open question** — see companion doc.

## Security Considerations

1. **Versioning integrity.** Supersession links (`supersedes`/`superseded_by`) form an append-only chain; superseding does not delete history.
2. **Prerequisite cycles.** Prevented off-chain; on-chain stores references only — consumers must not assume acyclicity from chain state alone.
3. **Append-only CFs.** No CF reorder/rename; new fields added additively with serde defaults.
4. **No content on-chain.** Description/syllabus/outcomes/accreditation are SNIP objects; chain holds commitments + policy only.

## Example Usage (illustrative, design-level)

```text
1. Registrar submits CreateCatalogEntry { department:"CS", course_code:"CS101",
   course_title:"Intro to CS", course_level:Undergraduate, credit_hours:3 }
   → CatalogEntry(status=Draft)
2. Registrar submits PublishCatalogContent { description_ref:(SnipRef, policy=Public),
   default_syllabus_ref:(SnipRef, policy=Public) } → status=Active
3. SRC-818 CreateOffering references this catalog_id (must be Active/Deprecated, not Archived)
4. Curriculum revision: SupersedeCatalogEntry → CatalogEntry v2 (Active), v1 Deprecated
```

## Open Questions

Consolidated and owner-tagged in [SRC-81X-EDUCATION-SUITE.md](SRC-81X-EDUCATION-SUITE.md) §Consolidated Legal / Product Questions. Catalog-specific open item: institution identity model (opaque `institution_id` commitment vs promotion to a first-class SRC-802 issuer).

**Recorded decision (locked):** plaintext `course_title` and `credit_hours` are the accepted defaults (verifiability for offerings / transcript / transfer-credit); `title_commitment` / `credit_commitment` remain available for confidential executive/proprietary programs. This is no longer an open question.
